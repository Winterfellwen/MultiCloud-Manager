Approach A: Minimal — Selective Auth

Minimal changeEasy to forget

Keep passthrough middleware, add LoginRequired per-route or per-group. Public routes exist alongside protected.

flowchart TB
    subgraph "router setup"
        AuthMW["AuthMiddleware(secret)\n= c.Next() (passthrough)"]
        Public["/api/auth/login\n/api/health"]
        Protected["everything else"]
    end
    subgraph "per-group"
        A["/api/accounts"]
        B["/api/resources"]
        C["/api/agent"]
    end
    Client --> Public
    Client --> AuthMW
    AuthMW --> A
    A --> LoginReq["accounts.Use(LoginRequired)\nvalidates JWT"]
    AuthMW --> B
    B --> LoginReq2["resources.Use(LoginRequired)"]
    style LoginReq fill:#3a2a1a,stroke:#d29922
    style LoginReq2 fill:#3a2a1a,stroke:#d29922
ProCon
Minimal code changes to existing routesEasy to forget LoginRequired on new routes
Good for quick iteration / prototypingRedundant JWT validation code in every group
Existing routes keep working unchangedTwo middleware layers confusing to debug

Approach B: Middleware Split — Public vs Protected

✅ RECOMMENDED
CleanOne auth surface

Restructure routes into two groups: public (auth, health) and protected (everything else). One real auth middleware on protected routes. RBAC layer on top for admin/operator restriction.

flowchart TB
    subgraph "Public (no auth)"
        Login["POST /api/auth/login\nusername/password → JWT"]
        WechatLogin["POST /api/auth/wechat\nwx.code → openid → JWT"]
        Health["/api/health"]
    end
    subgraph "Protected (real JWT check)"
        Auth["AuthMiddleware\nparse Bearer token\nverify JWT, extract user\nset c.Set('user', claims)\nreject if invalid"]
        RBAC["RBACMiddleware(roles...)\ncheck user.role ∈ allowed\n403 if unauthorized"]
        subgraph "Route groups"
            Accounts["/api/accounts"]
            Resources["/api/resources"]
            Agent["/api/agent"]
            Admin["/api/admin/users"]
        end
    end
    Client --> Public
    Client --> Auth
    Auth --> RBAC
    RBAC --> Accounts
    RBAC --> Resources
    RBAC --> Agent
    RBAC --> Admin
    style Login fill:#1a3a1a,stroke:#3fb950
    style Auth fill:#1a2a3a,stroke:#58a6ff
    style RBAC fill:#3a2a1a,stroke:#d29922
ProCon
One auth middleware, clear responsibilityNeed to define public-path exemption list
RBAC middleware reusable per groupRestructuring routes breaks if new routes added incorrectly
User context in gin.Context for handlersMigration needed for existing openid=NOT NULL constraint

Approach C: Dual Auth — Anonymous + Authenticated

FlexibleComplex

Single auth middleware that marks requests as authenticated or anonymous, never rejects. RBAC middleware rejects for protected routes. Enables "soft auth" for features that vary by login status.

flowchart TB
    subgraph "Single middleware chain"
        Auth["AuthMiddleware\ntry parse JWT\nset c.Set('user', claims) OR\nset c.Set('user', anonymous)\nalways call c.Next()"]
        RBAC["RBACMiddleware(requiredRole...)\nif user is anonymous and\nroute needs auth → 401\nif user.role ∉ allowed → 403"]
    end
    subgraph "Routes"
        Public["/api/auth/*"]
        Filtered["GET /api/stats\nshows different data based on auth"]
        Protected["/api/accounts\nneeds any valid login"]
        Admin["/api/admin/*\nneeds admin role"]
    end
    Client --> Auth
    Auth --> RBAC
    RBAC --> Public
    RBAC --> Filtered
    RBAC --> Protected
    RBAC --> Admin
    style Filtered fill:#1a2a3a,stroke:#58a6ff
ProCon
Supports "anonymous browsing" patternsMost routes don't need this — adds complexity
Single auth handler, no route-group splitAnonymous still hits auth processing (wasted work)
Good for public pages with auth-enhanced featuresThis project is all-protected (after login), so overkill

Data Flow — Login Sequence (Approach B)

sequenceDiagram
    actor U as User
    participant MP as Miniprogram
    participant API as Backend (Gin)
    participant DB as PostgreSQL
    U->>MP: Enter username/password
    MP->>API: POST /api/auth/login {username, password}
    API->>API: Validate input
    API->>DB: SELECT * FROM users WHERE username = ?
    DB-->>API: user row (with password_hash)
    API->>API: bcrypt.CompareHashAndPassword
    alt valid password
        API->>API: Generate JWT (sub=user.id, role=user.role, username)
        API-->>MP: {token, user: {id, username, nickname, role}}
        MP->>MP: Store token in storage
        MP->>MP: Redirect to home
    else invalid
        API-->>MP: 401 {error: "invalid credentials"}
        MP->>U: Show error
    end
    Note over U,DB: WeChat flow stays separate:
POST /api/auth/wechat → same JWT format
